# Handoff Document — nopCommerce Playwright Framework

## Context

This handoff carries forward lessons, patterns, and preferences from the Conduit (RealWorld) Playwright framework. The user is **learning** Playwright — they want coaching and guidance on how to write tests, not code written for them unless explicitly asked. Use 2026 best practices.

## Target Application

**nopCommerce Demo Store:** https://demo.nopcommerce.com/
- Full ecommerce platform (ASP.NET-based)
- Resets to original state every hour — no persistent test data (reinforces proper test isolation)
- Admin panel available at https://admin-demo.nopcommerce.com/
- Features: user registration/login, product search, filtering by category/color/price, sorting, product comparison, wishlist, cart, multi-step checkout, currency switching, contact forms, product tags, user account management, newsletter subscription, product reviews

### Config Patterns
- **tsconfig.json** — Strict mode, ESNext, NodeNext resolution, `@pages/*` path alias
- **playwright.config.ts:**
  - `dotenv` to load `.env` file
  - `baseURL` from `process.env.BASE_URL`
  - `workers: process.env.CI ? 1 : 2` (not unlimited — tests sharing state will collide)
  - `retries: process.env.CI ? 2 : 1`
  - Project separation: `setup`, `authenticated`, `unauthenticated`
  - `storageState` for authenticated project
- **Environment variables** — `.env` locally, GitHub Secrets in CI. Access via `process.env.VAR_NAME!` (non-null assertion for TypeScript)
- **CI/CD** — GitHub Actions workflow, env vars passed via `env` block on the test step (not workflow root)

### Page Object Pattern
- Locators as `readonly` class properties, defined in constructor
- `getByRole()` selectors throughout (accessibility-first, Playwright's #1 recommended approach)
- Methods for actions only — `goto()`, `fill*()`, `click*()`, etc.
- **Assertions stay in spec files**, never in page objects
- One page object per page, not per action
- `goto()` uses direct URL navigation for test setup; UI click-based navigation only when testing navigation itself

### Test Patterns
- `beforeAll` for expensive setup (creating test data via API)
- `beforeEach` for per-test state reset (via API, not UI)
- Independent test data per test — each test creates its own data so tests don't depend on each other
- `test.describe` blocks to scope preconditions
- `test.describe.configure({ mode: "serial" })` when tests mutate shared state
- **Proof of success** pattern: URL assertion + unique element visibility (1-2 strong assertions, not a wall of weak ones)
- Custom error messages on API assertions: `expect(response.ok(), \`Failed: ${response.status()} ${await response.text()}\`).toBeTruthy()`
- Token/auth-invalidating tests must run last in the file
- Never read files at module level that are generated by setup — always read inside `beforeAll`

### Fixture Pattern
```typescript
import { test as base } from "@playwright/test";
import { LoginPage } from "../pages/login.page";

type MyFixtures = {
  loginPage: LoginPage;
};

export const test = base.extend<MyFixtures>({
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await use(loginPage);
  },
});

export { expect } from "@playwright/test";
```

### Fake Data Pattern
- `@faker-js/faker` for generating test data
- Exported generator functions with TypeScript `ReturnType<typeof fn>` for type inference
- Timestamp-based unique identifiers for usernames/emails

## Key Lessons Learned (From Conduit)

### What Worked
- API-driven setup/teardown is faster and more reliable than UI-based setup
- `getByRole()` selectors are resilient to UI changes
- `page.evaluate()` to inject auth tokens into localStorage
- Waiting for form pre-fill before editing: `await expect(field).toHaveValue(oldValue)` before `fill()` — prevents race conditions
- Regex for locators with icon/invisible characters: `{ name: /Edit Article/ }`
- `.first()` for duplicate elements (resolves strict mode errors)
- API-based verification when UI can't verify (e.g., masked password fields)
- `beforeEach` API reset to prevent test interference
- Serial mode for tests that mutate shared user state

### What Didn't Work
- Reading generated files (like auth tokens) at module level — causes ENOENT in CI where setup hasn't run yet
- Exact string matching when hidden characters exist in text — use regex
- `fill()` without waiting for form to load — new value gets overwritten by late-loading old value
- Unlimited parallel workers with tests that share state — causes flaky failures
- `.env` file in CI — must use GitHub Secrets via `env` block on the test step
- `expect(response.ok()).toBeTruthy()` without a message — gives unhelpful failure output on API errors

## nopCommerce-Specific Considerations

### Auth Strategy
- nopCommerce uses cookie-based auth (unlike Conduit's JWT localStorage). Investigate whether `storageState` captures cookies correctly — it should, since Playwright's storageState includes cookies by default
- Check if nopCommerce has a public API for creating users/data, or if setup must be UI-based
- Admin panel login: admin@yourStore.com / admin (default demo credentials)

### Suggested Test Priority (Build Incrementally)
1. **Register + Login** — establish auth setup first
2. **Product browsing** — search, categories, filtering, sorting (new patterns!)
3. **Cart management** — add/remove items, quantity changes, price assertions
4. **Wishlist** — add/remove, move to cart
5. **Checkout flow** — multi-step (billing → shipping → payment → confirm) — most complex flow
6. **Product reviews** — write/read reviews
7. **User account** — order history, address book, change password
8. **Product comparison** — compare feature
9. **Admin panel** — separate project or test suite

### New Patterns to Learn
- **Price assertions** — parsing currency strings, asserting calculations
- **Multi-step forms** — checkout funnel with multiple pages
- **Dynamic filtering** — URL params changing, content updating without full page reload
- **Dropdown/select interactions** — currency selector, sort-by, category navigation
- **Table/grid assertions** — order history, product lists with multiple columns
