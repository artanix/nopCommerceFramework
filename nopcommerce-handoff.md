### Config Patterns
- **tsconfig.json** вЂ” Strict mode, ESNext, NodeNext resolution, `@pages/*` path alias
- **playwright.config.ts:**
  - `dotenv` to load `.env` file
  - `baseURL` from `process.env.BASE_URL`
  - `workers: process.env.CI ? 1 : 2` (not unlimited вЂ” tests sharing state will collide)
  - `retries: process.env.CI ? 2 : 1`
  - Project separation: `setup`, `authenticated`, `unauthenticated`
  - `storageState` for authenticated project
- **Environment variables** вЂ” `.env` locally, GitHub Secrets in CI. Access via `process.env.VAR_NAME!` (non-null assertion for TypeScript)
- **CI/CD** вЂ” GitHub Actions workflow, env vars passed via `env` block on the test step (not workflow root)

### Page Object Pattern
- Locators as `readonly` class properties, defined in constructor
- `getByRole()` selectors throughout (accessibility-first, Playwright's #1 recommended approach)
- Methods for actions only вЂ” `goto()`, `fill*()`, `click*()`, etc.
- **Assertions stay in spec files**, never in page objects
- One page object per page, not per action
- `goto()` uses direct URL navigation for test setup; UI click-based navigation only when testing navigation itself

### Test Patterns
- `beforeAll` for expensive setup (creating test data via API)
- `beforeEach` for per-test state reset (via API, not UI)
- Independent test data per test вЂ” each test creates its own data so tests don't depend on each other
- `test.describe` blocks to scope preconditions
- `test.describe.configure({ mode: "serial" })` when tests mutate shared state
- **Proof of success** pattern: URL assertion + unique element visibility (1-2 strong assertions, not a wall of weak ones)
- Custom error messages on API assertions: `expect(response.ok(), \`Failed: ${response.status()} ${await response.text()}\`).toBeTruthy()`
- Token/auth-invalidating tests must run last in the file
- Never read files at module level that are generated by setup вЂ” always read inside `beforeAll`

### Fixture Pattern
```typescript
import { test as base } from "@playwright/test";
import { LoginPage } from "../pages/login.page";

type MyFixtures = {
  loginPage: LoginPage;
};

export const test = base.extend<MyFixtures>({
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await use(loginPage);
  },
});

export { expect } from "@playwright/test";
```

### Fake Data Pattern
- `@faker-js/faker` for generating test data
- Exported generator functions with TypeScript `ReturnType<typeof fn>` for type inference
- Timestamp-based unique identifiers for usernames/emails

## Key Lessons Learned (From Conduit)

### What Worked
- API-driven setup/teardown is faster and more reliable than UI-based setup
- `getByRole()` selectors are resilient to UI changes
- `page.evaluate()` to inject auth tokens into localStorage
- Waiting for form pre-fill before editing: `await expect(field).toHaveValue(oldValue)` before `fill()` вЂ” prevents race conditions
- Regex for locators with icon/invisible characters: `{ name: /Edit Article/ }`
- `.first()` for duplicate elements (resolves strict mode errors)
- API-based verification when UI can't verify (e.g., masked password fields)
- `beforeEach` API reset to prevent test interference
- Serial mode for tests that mutate shared user state

### What Didn't Work
- Reading generated files (like auth tokens) at module level вЂ” causes ENOENT in CI where setup hasn't run yet
- Exact string matching when hidden characters exist in text вЂ” use regex
- `fill()` without waiting for form to load вЂ” new value gets overwritten by late-loading old value
- Unlimited parallel workers with tests that share state вЂ” causes flaky failures
- `.env` file in CI вЂ” must use GitHub Secrets via `env` block on the test step
- `expect(response.ok()).toBeTruthy()` without a message вЂ” gives unhelpful failure output on API errors

## nopCommerce-Specific Considerations

### Auth Strategy
- nopCommerce uses cookie-based auth (unlike Conduit's JWT localStorage). `storageState` captures cookies by default вЂ” this should work
- No public API вЂ” setup must be UI-based
- Admin panel login: admin@yourStore.com / (password set during Docker install)

### Suggested Test Priority (Build Incrementally)
1. **Register + Login** вЂ” establish auth setup first
2. **Product browsing** вЂ” search, categories, filtering, sorting (new patterns!)
3. **Cart management** вЂ” add/remove items, quantity changes, price assertions
4. **Wishlist** вЂ” add/remove, move to cart
5. **Checkout flow** вЂ” multi-step (billing в†’ shipping в†’ payment в†’ confirm) вЂ” most complex flow
6. **Product reviews** вЂ” write/read reviews
7. **User account** вЂ” order history, address book, change password
8. **Product comparison** вЂ” compare feature
9. **Admin panel** вЂ” separate project or test suite

### New Patterns to Learn
- **Price assertions** вЂ” parsing currency strings, asserting calculations
- **Multi-step forms** вЂ” checkout funnel with multiple pages
- **Dynamic filtering** вЂ” URL params changing, content updating without full page reload
- **Dropdown/select interactions** вЂ” currency selector, sort-by, category navigation
- **Table/grid assertions** вЂ” order history, product lists with multiple columns
